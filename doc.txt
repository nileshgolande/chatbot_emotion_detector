================================================================================
1. WHAT THIS PROJECT IS
================================================================================
  A full-stack “emotion-aware” chat product:

  • Frontend (React): Landing, login/register, WhatsApp-style chat UI, mood
    dashboard (Recharts), journal views. Uses JWT in localStorage; axios client
    with token refresh on 401.

  • Backend (Django + DRF): REST API for auth, conversations/messages, emotion
    analysis on each user message, LLM replies (Groq / OpenRouter / Gemini),
    LangGraph wrapper for chat orchestration, journal digests, dashboard
    aggregates. PostgreSQL + pgvector for optional 768-d embeddings on
    EmotionAnalysis.

  • Optional local demo mode: guest “demo” user in browser without API tokens;
    replies are stubbed on the client.

2. ARCHITECTURE (HOW PIECES CONNECT)
================================================================================

  Browser (React)
       |  HTTP JSON + Bearer JWT
       v
  Django ASGI (HTTP via DRF; WebSocket routing file exists but pattern list empty)
       |
       +-- /api/auth/*     → accounts (register, JWT, profile me/)
       +-- /api/chat/*     → chat (conversations CRUD, send_message, llm/)
       +-- /api/emotions/* → emotions summary
       +-- /api/journal/*  → journal entries + digest + quick_reflect
       +-- /api/dashboard/*→ mood timeline, trends, reports

  Chat reply path (simplified):
    User POST → ConversationViewSet.send_message (or create with first_message)
      → _send_message_payload()
         → save Message (user)
         → emotion_detector.detect_emotion() → EmotionAnalysis row
         → transaction.on_commit → background thread update_daily_mood(user)
         → extract_and_merge_memory(user, text) → updates UserChatMemory (name
           heuristics, “remember that …” facts, message counts) in Postgres
         → memory_ctx = build_llm_user_context(user) → formatted memory block +
           DB “account snapshot” (conversation counts, message totals, latest mood)
         → last 5 prior messages as history (current thread only)
         → run_chat_graph(text, emotion, history, memory_context=memory_ctx)
              → LangGraph: prepare_context passes memory_context through state
              → generate node → get_llm_response(..., memory_context=...)
              → build_chat_system_prompt(primary_emotion, memory_context) appends
                 user-specific context before hard rules
              → LLM_ORDER providers in llm_service (Gemini / Groq / OpenRouter
                 with multi-key and multi-model fallbacks; see §4.5)
         → save Message (bot)
         → return serialized messages + conversation

  Same memory injection applies to GET/POST /api/chat/llm/ (llm_response_view):
    detect_emotion → extract_and_merge_memory → build_llm_user_context →
    run_chat_graph(..., memory_context=...) (no conversation Message row there).

4.4 chat/
----------
  apps.py
    • ChatConfig.ready() imports chat.signals so receivers register.

  signals.py
    • post_save User → if created: Conversation(title=WELCOME_TITLE) +
      Message(sender=bot, WELCOME_BOT_MESSAGE). Seeds first chat for new users.

  models.py
    • Conversation: FK user, title, JSON agent_metadata, relationship stage,
      floats empathy_effectiveness / user_satisfaction, timestamps. Ordered by
      -created_at.
    • Message: FK conversation, sender user|bot, content, timestamps. Ordered
      by created_at.
    • UserChatMemory: OneToOne user — preferred_name, facts JSON, recent_emotions,
      user_message_count, updated_at. Maintained by chat/user_memory.py
      (extract_and_merge_memory) from user text patterns; read back into the LLM
      each turn via build_llm_user_context (cross-thread “one companion” behavior).

  user_memory.py
    • extract_and_merge_memory(user, message_text): after each user turn, updates
      UserChatMemory (get_or_create) — increments counts, parses phrases like
      “my name is / call me / I’m …” for preferred_name, “remember that …” lines
      into capped facts list.
    • format_memory_for_prompt(mem): turns DB memory into system-prompt-ready text.
    • build_account_snapshot(user): aggregates from DB (username, date_joined,
      conversation count, total messages, user-sent count, latest emotion label).
    • build_llm_user_context(user): joins memory block + snapshot for the model
      (answers “what’s my name?”, “what data do you have?”, stats-style questions).

  serializers.py
    • MessageSerializer: id, sender, content, emotion (SerializerMethodField),
      created_at. get_emotion: only for sender=user; reads OneToOne emotion
      (EmotionAnalysis) or null; adds emoji from emotions.emotion_emojis.
    • ConversationSerializer: id, title, last_message_preview (from latest
      message text, max 120 chars), dates.
    • ConversationDetailSerializer: adds nested messages list + agent stage.
    • ConversationCreateSerializer: title + write_only first_message;
      create() pops first_message (handled in view).

  daily_mood.py
    • update_daily_mood(user, analysis=None): aggregates today’s
      EmotionAnalysis for user → dominant emotion, distribution, avg confidence,
      count → DailyMoodSummary.update_or_create. If no rows today, deletes
      summary. “analysis” arg ignored (API compat).

  empathy_prompts.py
    • empathy_core_instructions(): long system-style string (tone, length, emoji
      discipline, no therapy-speak).
    • chat_api_hard_rules(): no meta “You shared” lines, reply-context rule,
      safety.
    • emotion_guidance(emotion): short bullet per mood bucket.
    • build_chat_system_prompt(primary_emotion, memory_context=None): concatenates
      empathy + emotion hint + optional “User-specific context …” block (memory
      + account snapshot) + hard rules. Used by llm_service.get_llm_response.

  langgraph_chat.py
    • ChatGraphState (TypedDict): user_input, primary_emotion, history,
      memory_context, reply.
    • _USE_GRAPH from env USE_LANGGRAPH_CHAT (default on).
    • _node_prepare_context: normalizes emotion string; ensures history list;
      passes through memory_context for the LLM.
    • _node_generate: calls services.llm_service.get_llm_response with
      memory_context from state.
    • _build_graph / _compiled_graph: StateGraph START→prepare→generate→END.
    • run_chat_graph(..., memory_context=None): if graph disabled, direct
      get_llm_response(with memory_context); else invoke compiled graph with
      memory_context in initial state; empty reply falls back to get_llm_response.

  views.py
    • _send_message_payload(request, conv, content): core chat transaction
      helper — validates content, creates user Message, maybe sets title from
      first user line, detect_emotion, EmotionAnalysis.create, schedules
      update_daily_mood after commit, extract_and_merge_memory +
      build_llm_user_context, builds history (5 msgs excluding current), run_chat_graph(…,
      memory_context=…), bot Message, returns serialized messages + conversation.
    • _title_from_first_message: whitespace-normalized title ≤120 chars.
    • llm_response_view (GET/POST /api/chat/llm/): bare “ask LLM” for auth
      users; emotion detect + extract_and_merge_memory + build_llm_user_context +
      run_chat_graph(..., memory_context=...) (no conversation-scoped history).
    • ConversationViewSet:
        - List/retrieve/destroy user’s conversations.
        - create: optional atomic path with first_message → _send_message_payload.
        - messages GET: paginated MessageSerializer list.
        - send_message POST: atomic _send_message_payload.

  urls.py
    • DefaultRouter registers conversations.
    • Path llm/ → llm_response_view.

  admin.py — registers models for Django admin.

  management/commands/seed_demo_data.py
    • Large QA command: creates/replaces demo User + UserProfile + many
      Conversations, Messages, EmotionAnalysis, DailyMoodSummary, JournalEntry,
      JournalInsights with spread over SEED_DEMO_HISTORY_DAYS. Uses helpers
      like _env_int, fill loops — read file for env knobs (SEED_DEMO_USERNAME…).

4.5 services/llm_service.py
----------------------------
  Responsibility: “bulletproof” multi-provider chat — raw HTTP via requests, no
  LangChain client classes. Tries multiple API keys per provider (GEMINI_API_KEY,
  GEMINI_API_KEY_2…, same pattern for GROQ_*, OPENROUTER_*) and multiple model IDs
  per provider until something succeeds.

  Provider order: env LLM_ORDER (comma-separated: gemini, groq, openrouter).
  Default in code is gemini → groq → openrouter (override in .env e.g.
  groq,openrouter,gemini for lower latency when Groq quota is healthy).

  Model lists (edit in module or extend logic as needed):
    • GEMINI_FREE_MODELS — tries each model × each Gemini key × API versions v1beta/v1
    • GROQ_FREE_MODELS — tries each model id with each Groq key (OpenAI-compatible URL)
    • OPENROUTER_FREE_MODELS — long :free-tier list; some endpoints retire often

  Reliability helpers:
    • _classify_error(status, body): rate limit vs expired key vs bad model vs
      bad request (e.g. provider rejects separate system role)
    • _merge_system_into_user(messages): merges system instructions into the first
      user turn — used by OpenRouter path when a normal system message fails
    • call_openrouter_chat: for each key, for each model, tries original messages
      then merged variant when applicable
    • call_gemini(prompt), call_groq_chat(messages): rotate keys/models internally
    • _build_chat_messages, _messages_to_gemini_prompt: chat transcript + system
    • _sanitize_llm_text: strips “(You shared: …)” style meta from outputs
    • get_llm_response(user_input, primary_emotion, history, memory_context=None):
      builds system prompt via build_chat_system_prompt(primary_emotion,
      memory_context); runs provider chain; optionally retries reverse order;
      on total failure logs loudly and returns EMOTION_FALLBACK_MESSAGES[emotion]
    • complete_chat_messages(messages, max_tokens?): same provider routing for
      journal — journal_ai_insights.JournalAIInsights

  Operational notes: if every provider fails (expired Gemini key, Groq daily TPD
  429, OpenRouter “no endpoints” / free-tier limits), the API still returns 200
  with short “technical hiccup” fallback text — fix keys and quotas, watch
  server logs for HTTP status lines.

  Connection: chat/langgraph_chat (generate node), journal/journal_ai_insights,
  emotions/emotion_detector (optional Gemini refine when EMOTION_LLM_REFINE=1).

  File hygiene: if you see a long block of # -commented legacy code above the
  active module docstring, delete everything above the opening """ docstring
  that precedes from __future__ import annotations so only one implementation
  remains (avoids confusion and accidental edits inside comments).

4.6 emotions/
--------------
  models.py
    • EmotionAnalysis: user FK, message OneToOne (user messages only), primary
      emotion, JSON scores, VectorField 768 optional, confidence, created_at.
    • DailyMoodSummary: user, date unique together, dominant_emotion,
      distribution JSON, avg_confidence, messages_count.

  emotion_detector.py (pipeline)
    • Module constants: EMOTIONS tuple, env thresholds EMOTION_FAST_KEYWORD_THRESHOLD,
      EMOTION_LLM_REFINE, EMOTION_COMPUTE_EMBEDDING (off by default = no 768 vec).
    • _get_embedder(): lazy SentenceTransformer all-mpnet-base-v2 (768-d).
    • _get_classifier(): Hugging Face pipeline j-hartmann/emotion-english-*.
    • _map_label_to_bucket: HF labels → happy/sad/anxious/angry/neutral.
    • _keyword_scores: regex-based fast scores.
    • _heuristic_refine_primary: e.g. frustrated text not labeled happy.
    • _llm_refine_primary_emotion: optional Gemini sanity-check when env on.
    • EmotionDetector.detect_emotion(text): keyword fast path OR classifier path;
      optional embedding array; returns dict for views → EmotionAnalysis.
    • emotion_detector: singleton instance.
    • warmup_emotion_classifier(): preload pipeline (optional startup).
    • analyze_text: alias to detect_emotion.

  emotion_emojis.py
    • EMOTION_EMOJIS dict for API/serializer display (keep aligned with frontend
      emotionChartTheme.js comment).

  views.py
    • EmotionSummaryView GET: total analyses, by_emotion counts, latest row.

  urls.py — summary/

7. HOW TO EXTEND
================================================================================
  • More LangGraph nodes: edit chat/langgraph_chat.py — add retrieval from
    pgvector, tool nodes, conditional edges — then call same get_llm_response
    (pass memory_context through state) or new chains.
  • Richer memory: extend user_memory.py (more extractors, summarization jobs) or
    feed retrieved Message snippets into memory_context with a size cap.
  • Real WebSockets: add consumers under a new app, append routes in
    emotion_chat/routing.py.
  • Vector search: enable EMOTION_COMPUTE_EMBEDDING and add queryset filters
    using pgvector distance in Django ORM.
