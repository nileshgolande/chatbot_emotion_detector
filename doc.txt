================================================================================
1. WHAT THIS PROJECT IS
================================================================================
  A full-stack “emotion-aware” chat product:

  • Frontend (React): Landing, login/register, WhatsApp-style chat UI, mood
    dashboard (Recharts), journal views. Uses JWT in localStorage; axios client
    with token refresh on 401.

  • Backend (Django + DRF): REST API for auth, conversations/messages, emotion
    analysis on each user message, LLM replies (Groq / OpenRouter / Gemini),
    LangGraph wrapper for chat orchestration, journal digests, dashboard
    aggregates. PostgreSQL + pgvector for optional 768-d embeddings on
    EmotionAnalysis.

  • Optional local demo mode: guest “demo” user in browser without API tokens;
    replies are stubbed on the client.

2. ARCHITECTURE (HOW PIECES CONNECT)
================================================================================

  Browser (React)
       |  HTTP JSON + Bearer JWT
       v
  Django ASGI (HTTP via DRF; WebSocket routing file exists but pattern list empty)
       |
       +-- /api/auth/*     → accounts (register, JWT, profile me/)
       +-- /api/chat/*     → chat (conversations CRUD, send_message, llm/)
       +-- /api/emotions/* → emotions summary
       +-- /api/journal/*  → journal entries + digest + quick_reflect
       +-- /api/dashboard/*→ mood timeline, trends, reports

  Chat reply path (simplified):
    User POST → ConversationViewSet.send_message (or create with first_message)
      → _send_message_payload()
         → save Message (user)
         → emotion_detector.detect_emotion() → EmotionAnalysis row
         → transaction.on_commit → background thread update_daily_mood(user)
         → last 5 prior messages as history
         → run_chat_graph(text, emotion, history)
              → LangGraph: prepare_context → generate node → get_llm_response()
              → LLM_ORDER providers in llm_service
         → save Message (bot)
         → return serialized messages + conversation

4.4 chat/
----------
  apps.py
    • ChatConfig.ready() imports chat.signals so receivers register.

  signals.py
    • post_save User → if created: Conversation(title=WELCOME_TITLE) +
      Message(sender=bot, WELCOME_BOT_MESSAGE). Seeds first chat for new users.

  models.py
    • Conversation: FK user, title, JSON agent_metadata, relationship stage,
      floats empathy_effectiveness / user_satisfaction, timestamps. Ordered by
      -created_at.
    • Message: FK conversation, sender user|bot, content, timestamps. Ordered
      by created_at.
    • UserChatMemory: OneToOne user — preferred_name, facts JSON, recent_emotions,
      message count (extensible “memory” for future RAG/agents).

  serializers.py
    • MessageSerializer: id, sender, content, emotion (SerializerMethodField),
      created_at. get_emotion: only for sender=user; reads OneToOne emotion
      (EmotionAnalysis) or null; adds emoji from emotions.emotion_emojis.
    • ConversationSerializer: id, title, last_message_preview (from latest
      message text, max 120 chars), dates.
    • ConversationDetailSerializer: adds nested messages list + agent stage.
    • ConversationCreateSerializer: title + write_only first_message;
      create() pops first_message (handled in view).

  daily_mood.py
    • update_daily_mood(user, analysis=None): aggregates today’s
      EmotionAnalysis for user → dominant emotion, distribution, avg confidence,
      count → DailyMoodSummary.update_or_create. If no rows today, deletes
      summary. “analysis” arg ignored (API compat).

  empathy_prompts.py
    • empathy_core_instructions(): long system-style string (tone, length, emoji
      discipline, no therapy-speak).
    • chat_api_hard_rules(): no meta “You shared” lines, reply-context rule,
      safety.
    • emotion_guidance(emotion): short bullet per mood bucket.
    • build_chat_system_prompt(primary_emotion): concatenates the above for HTTP
      chat. Used by llm_service.get_llm_response.

  langgraph_chat.py
    • ChatGraphState (TypedDict): user_input, primary_emotion, history, reply.
    • _USE_GRAPH from env USE_LANGGRAPH_CHAT (default on).
    • _node_prepare_context: normalizes emotion string; ensures history list.
    • _node_generate: calls services.llm_service.get_llm_response.
    • _build_graph / _compiled_graph: StateGraph START→prepare→generate→END.
    • run_chat_graph(...): if graph disabled, direct get_llm_response; else
      invoke compiled graph; empty reply falls back to get_llm_response.

  views.py
    • _send_message_payload(request, conv, content): core chat transaction
      helper — validates content, creates user Message, maybe sets title from
      first user line, detect_emotion, EmotionAnalysis.create, schedules
      update_daily_mood after commit, builds history (5 msgs excluding current),
      run_chat_graph, bot Message, returns serialized messages + conversation.
    • _title_from_first_message: whitespace-normalized title ≤120 chars.
    • llm_response_view (GET/POST /api/chat/llm/): bare “ask LLM” for auth
      users; emotion detect + run_chat_graph (no history in this endpoint).
    • ConversationViewSet:
        - List/retrieve/destroy user’s conversations.
        - create: optional atomic path with first_message → _send_message_payload.
        - messages GET: paginated MessageSerializer list.
        - send_message POST: atomic _send_message_payload.

  urls.py
    • DefaultRouter registers conversations.
    • Path llm/ → llm_response_view.

  admin.py — registers models for Django admin.

  management/commands/seed_demo_data.py
    • Large QA command: creates/replaces demo User + UserProfile + many
      Conversations, Messages, EmotionAnalysis, DailyMoodSummary, JournalEntry,
      JournalInsights with spread over SEED_DEMO_HISTORY_DAYS. Uses helpers
      like _env_int, fill loops — read file for env knobs (SEED_DEMO_USERNAME…).

4.5 services/llm_service.py
----------------------------
  Responsibility: multi-provider LLM calls with env-driven LLM_ORDER (default
  groq → openrouter → gemini). No LangChain client classes — raw HTTP via
  requests where applicable.

  Key helpers (non-exhaustive internals):
    • _strip_env_value, gemini_api_key, groq_api_key, openrouter_api_key
    • _invalid_*_key: rejects empty/placeholder keys with “...”
    • gemini_model_candidates: parses GEMINI_MODEL comma list
    • _safe_json: response.json() with logging
    • _sanitize_llm_text: strips “(You shared: …)” style meta from model output
    • _build_chat_messages(user_input, history, system_prompt): OpenAI-style
      message list with last 5 history turns mapped user/assistant from Message
      or dict
    • _call_gemini_one_model, _messages_to_gemini_prompt, call_gemini
    • call_groq_chat(messages, max_tokens?), call_groq(single user string)
    • call_openrouter_chat: tries OPENROUTER_MODELS list
    • complete_chat_messages(messages, max_tokens?): same order as chat but for
      journal — used by journal_ai_insights.JournalAIInsights
    • get_llm_response(user_input, primary_emotion, history): builds system prompt
      via build_chat_system_prompt; loops providers; on total failure returns
      EMOTION_FALLBACK_MESSAGES[primary]

  Connection: imported by chat/langgraph_chat (generate node), journal
  journal_ai_insights, emotions emotion_detector (optional call_gemini for refine).

4.6 emotions/
--------------
  models.py
    • EmotionAnalysis: user FK, message OneToOne (user messages only), primary
      emotion, JSON scores, VectorField 768 optional, confidence, created_at.
    • DailyMoodSummary: user, date unique together, dominant_emotion,
      distribution JSON, avg_confidence, messages_count.

  emotion_detector.py (pipeline)
    • Module constants: EMOTIONS tuple, env thresholds EMOTION_FAST_KEYWORD_THRESHOLD,
      EMOTION_LLM_REFINE, EMOTION_COMPUTE_EMBEDDING (off by default = no 768 vec).
    • _get_embedder(): lazy SentenceTransformer all-mpnet-base-v2 (768-d).
    • _get_classifier(): Hugging Face pipeline j-hartmann/emotion-english-*.
    • _map_label_to_bucket: HF labels → happy/sad/anxious/angry/neutral.
    • _keyword_scores: regex-based fast scores.
    • _heuristic_refine_primary: e.g. frustrated text not labeled happy.
    • _llm_refine_primary_emotion: optional Gemini sanity-check when env on.
    • EmotionDetector.detect_emotion(text): keyword fast path OR classifier path;
      optional embedding array; returns dict for views → EmotionAnalysis.
    • emotion_detector: singleton instance.
    • warmup_emotion_classifier(): preload pipeline (optional startup).
    • analyze_text: alias to detect_emotion.

  emotion_emojis.py
    • EMOTION_EMOJIS dict for API/serializer display (keep aligned with frontend
      emotionChartTheme.js comment).

  views.py
    • EmotionSummaryView GET: total analyses, by_emotion counts, latest row.

  urls.py — summary/

4.7 journal/
--------------
  models.py
    • JournalEntry: user, title, content, mood_at_entry, ai_insights text,
      tags JSON, timestamps.
    • JournalInsights: user, period weekly|monthly, date range, JSON top_emotions,
      text patterns/recommendations, sentiment_trend JSON; unique on
      (user, period_type, period_start).

  daily_digest.py
    • user_display_label(user): first+last name or username for LLM copy.
    • _safe_pct, _wellness_score, _dominant: stats helpers from per-day counts.
    • _day_emotion_counts, _day_timeline: query EmotionAnalysis + message snippets.
    • _activity_streak_ending, _streak_calendar: streak UX data.
    • _heuristic_insights, _fallback_story: non-LLM narrative pieces.
    • build_daily_digest(user, target_date): large JSON “story” for journal UI
      (counts, timeline, wellness, insights cards, etc.).
    • apply_llm_story_patch(payload, raw_llm): merges JSON from LLM into digest
      when enrich_daily_digest returns structured text.

  journal_ai_insights.py
    • JournalAIInsights.generate_entry_insights(title, content, mood, name):
      system+user messages → complete_chat_messages → sanitize.
    • generate_period_insights: titles + emotion summary → LLM → patterns +
      recommendations dict.
    • enrich_daily_digest(payload): slim JSON to LLM → apply_llm_story_patch.
    • _journal_offline_hint: user-facing string when no provider works.
    • journal_ai: module singleton.

  serializers.py — JournalEntrySerializer, JournalInsightsSerializer (fields,
    read_only on ai_insights / insights).

  views.py
    • JournalEntryViewSet: CRUD scoped to user; today list;
      daily_digest (build_daily_digest + journal_ai.enrich_daily_digest);
      quick_reflect POST (keyword mood guess + generate_entry_insights);
      generate_insights POST on entry pk (persists ai_insights).
    • JournalInsightsViewSet: read-only list/detail for user.

  urls.py — routers entries/, insights/

7. HOW TO EXTEND
================================================================================
  • More LangGraph nodes: edit chat/langgraph_chat.py — add retrieval from
    pgvector, tool nodes, conditional edges — then call same get_llm_response
    or new chains.
  • Real WebSockets: add consumers under a new app, append routes in
    emotion_chat/routing.py.
  • Vector search: enable EMOTION_COMPUTE_EMBEDDING and add queryset filters
    using pgvector distance in Django ORM.
